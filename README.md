# backlog-github-sync

One-way sync from [Nulab Backlog](https://backlog.com) to [GitHub Projects V2](https://docs.github.com/en/issues/planning-and-tracking-with-projects). Runs as a GitHub Action on a 5-minute cron schedule, creating and updating draft items in your project board.

## How It Works

1. Polls Backlog API for tickets updated since last sync
2. Creates **draft items** in a GitHub Project V2 board (no GitHub Issues)
3. Maps Backlog fields (status, priority, assignee, etc.) to Project custom fields
4. Tracks sync state in `.sync-state.json` (committed to repo)

## Setup

### 1. Create GitHub Project V2 Custom Fields

In your GitHub Project, create these custom fields (names must match `config/field-mapping.yml`):

| Field Name | Type | Options |
|---|---|---|
| Status | Single Select | Todo, In Progress, Done |
| Priority | Single Select | High, Medium, Low |
| Backlog Assignee | Text | - |
| Due Date | Date | - |
| Backlog Key | Text | - |
| Backlog URL | Text | - |

### 2. Get a Backlog API Key

1. Log in to Backlog
2. Personal Settings -> API -> Register new API key
3. Note your **space name** and **project ID** (numeric, found in API settings or project URL)

### 3. Get a GitHub PAT

The default `GITHUB_TOKEN` does **not** have `project` scope. Create a PAT:

**Classic PAT:** Scopes: `repo`, `project`

**Fine-grained PAT:** Repository access + Projects read/write

### 4. Configure Repository Secrets & Variables

**Secrets** (Settings -> Secrets and variables -> Actions -> Secrets):

| Secret | Value |
|---|---|
| `BACKLOG_SPACE` | Your Backlog space name (e.g., `mycompany`) |
| `BACKLOG_DOMAIN` | `backlog.com` or `backlog.jp` |
| `BACKLOG_API_KEY` | Your Backlog API key |
| `BACKLOG_PROJECT_ID` | Backlog project ID (numeric) |
| `PROJECT_PAT` | GitHub PAT with `repo` + `project` scopes |

**Variables** (Settings -> Secrets and variables -> Actions -> Variables):

| Variable | Value |
|---|---|
| `PROJECT_OWNER` | GitHub user or org that owns the project |
| `PROJECT_NUMBER` | Project number (from project URL) |
| `INITIAL_SYNC_HOURS` | (Optional) Hours to look back on first run (default: 720 = 30 days) |

### 5. Create Field Mapping Config

```bash
cp config/field-mapping.example.yml config/field-mapping.yml
```

Edit `config/field-mapping.yml` to match your Backlog statuses/priorities to your Project field options.

### 6. Enable the Workflow

The workflow runs automatically every 5 minutes. You can also trigger it manually:

Actions -> Backlog to GitHub Projects Sync -> Run workflow

## Configuration

### Field Mapping (`config/field-mapping.yml`)

Maps Backlog fields to Project V2 custom fields. Supported types:

- `single_select` â€” requires `value_map` to translate values
- `text` â€” direct string copy
- `date` â€” ISO date format
- `number` â€” numeric value

### Environment Variables

| Variable | Required | Default | Description |
|---|---|---|---|
| `BACKLOG_SPACE` | Yes | - | Backlog space name |
| `BACKLOG_DOMAIN` | No | `backlog.com` | `backlog.com` or `backlog.jp` |
| `BACKLOG_API_KEY` | Yes | - | Backlog API key |
| `BACKLOG_PROJECT_ID` | Yes | - | Backlog project ID |
| `GITHUB_TOKEN` | Yes | - | GitHub PAT with project scope |
| `GITHUB_PROJECT_OWNER` | Yes | - | Project owner (user/org) |
| `GITHUB_PROJECT_NUMBER` | Yes | - | Project number |
| `FIELD_MAPPING_PATH` | No | `config/field-mapping.yml` | Path to field mapping config |
| `INITIAL_SYNC_HOURS` | No | `720` | Hours to look back on first run |
| `MAX_ISSUES_PER_RUN` | No | `500` | Max tickets per sync cycle |
| `OVERLAP_BUFFER_MINUTES` | No | `1` | Overlap buffer to avoid missing updates |

## State File

`.sync-state.json` is auto-managed and committed by the workflow. It tracks:

- `lastPolledAt` â€” timestamp of last successful sync
- `projectItemMap` â€” mapping of Backlog issue keys to Project item IDs
- `projectFieldIds` â€” cached Project field IDs and options
